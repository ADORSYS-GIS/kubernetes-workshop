Container Runtime Landscape

Kubernetes supports multiple container engines through the Container Runtime Interface (CRI)

High-Level Runtimes

Docker Engine

Full-featured platform with daemon, CLI, and registry integration

containerd

Lightweight, core container runtime extracted from Docker

Kubernetes-native runtime designed specifically for CRI compliance

Industry standard low-level runtime (OCI compliant)

crun

Fast, lightweight alternative written in C

kata-containers

VM-based isolation with container interface

gVisor

Application kernel for enhanced security isolation

Firecracker

MicroVM technology for serverless and multi-tenant workloads

rkt

CoreOS container runtime (deprecated)

Anatomy of a Container

Containers aren't magic. They are built upon two core Linux kernel technologies that provide process isolation and resource management.

Namespaces: The Illusion of Isolation

Namespaces ensure that what happens in a container stays in a container. Each container gets its own isolated view of the system, including its own process tree (PID), network stack (NET), and filesystem mounts (MNT). It's like giving each application its own private operating system without the overhead of a full VM.

PIDIsolated process IDs. An app can be PID 1.
NETIsolated network interface, IP, and ports.
MNTIsolated filesystem view.

Cgroups: Setting Resource Limits

Control Groups (cgroups) are the "resource police" for containers. They enforce limits on how much CPU and Memory a container can consume, preventing a single "noisy neighbor" application from starving other applications on the same host machine. This is critical for system stability in a multi-tenant environment.

CPULimit usage to a specific share or number of cores.
MemorySet hard limits on RAM consumption.

The Challenge of Scale

Running one container is simple. Managing thousands across a fleet of servers requires a powerful orchestrator like Kubernetes.

Orchestration Responsibilities

An orchestrator automates the operational tasks required to run containerized applications resiliently. It's the "brain" that manages the entire lifecycle, from deployment and scaling to networking and self-healing, ensuring the desired state of your application is always maintained.

Kubernetes Architecture

A Kubernetes cluster is divided into a Control Plane (the brain) and a set of Worker Nodes (the muscle) that run your applications.

🧠 Control Plane (Master Nodes)

The brain of the cluster. Makes all scheduling decisions, stores cluster state, and provides the API that everything else talks to.

🚪 API Server

Role: The single point of entry for all cluster operations

Responsibilities:

  • Validates and processes REST API requests
  • Authenticates and authorizes users
  • Runs admission controllers
  • Interfaces with etcd for data persistence

Port: 6443 (HTTPS)

🗄️ etcd

Role: Distributed key-value store and source of truth

Responsibilities:

  • Stores all cluster data and configuration
  • Maintains cluster state consistency
  • Provides strong consistency guarantees
  • Supports cluster backup and restore

Port: 2379 (client), 2380 (peer)

📅 Scheduler

Role: Intelligent pod placement engine

Responsibilities:

  • Watches for unscheduled pods
  • Evaluates node resources and constraints
  • Applies scheduling policies and priorities
  • Assigns pods to optimal nodes

Algorithm: Filter → Score → Bind

🔄 Controller Manager

Role: Maintains desired state through reconciliation loops

Key Controllers:

  • Replication Controller
  • Deployment Controller
  • Service Controller
  • Node Controller
  • Endpoint Controller

Pattern: Watch → Compare → Act

🔒 High Availability Considerations

etcd Cluster: Usually 3 or 5 nodes for fault tolerance

API Server: Can run multiple instances behind a load balancer

Scheduler: Leader election ensures only one is active

Controller Manager: Also uses leader election

💪 Worker Nodes (The Muscle)

Where your applications actually run. Each worker node contains the necessary components to run pods and communicate with the control plane.

Worker Node Architecture
🤖 Kubelet

Role: Node agent that manages pods

  • Watches API Server for pod assignments
  • Ensures containers are running and healthy
  • Reports node and pod status back to API Server
  • Manages pod lifecycle and resource limits
🌐 Kube-Proxy

Role: Network proxy for Services

  • Maintains network rules for Service routing
  • Implements load balancing across pod endpoints
  • Handles iptables/IPVS rule management
  • Enables service discovery and connectivity
🐳 Container Runtime

Options: Docker, containerd, CRI-O

  • Pulls container images from registries
  • Creates and manages container lifecycle
  • Implements CRI (Container Runtime Interface)
  • Handles container networking and storage

Pod Creation Journey

What happens inside Kubernetes when you run kubectl apply -f pod.yaml? Follow the internal workflow from command to running container.

1. kubectl → API Server

POST /api/v1/namespaces/default/pods
1

1. kubectl → API Server

kubectl reads your YAML file and sends an HTTP POST request to the API Server with the pod specification.

2. API Server Validation

The API Server validates the pod spec against the schema, checks permissions, and runs admission controllers.

2

2. API Server Validation

Validate → Authenticate → Authorize

3. Store in etcd

etcd PUT /registry/pods/default/my-pod
3

3. Store in etcd

The API Server persists the pod object to etcd. The pod is now in "Pending" state with no assigned node.

4. Scheduler Watches

The Scheduler watches for unscheduled pods, evaluates node resources, and selects the best node for placement.

4

4. Scheduler Watches

Watch → Filter → Score → Bind

5. Update Pod Assignment

PATCH pod.spec.nodeName = "worker-node-1"
5

5. Update Pod Assignment

The Scheduler updates the pod object in etcd with the assigned node name through the API Server.

6. Kubelet Watches

The Kubelet on the assigned node watches for pods scheduled to it and pulls the pod specification.

6

6. Kubelet Watches

Watch pods where spec.nodeName = self

7. Container Runtime

docker pull nginx:latest
7

7. Container Runtime

Kubelet instructs the container runtime (Docker/containerd) to pull the image and create the container.

8. Status Update

Kubelet reports the pod status back to the API Server, which updates etcd. Pod status changes to "Running".

8

8. Status Update

PATCH pod.status.phase = "Running"

🎉 Container Running!

Your application is now running in a container on the Kubernetes cluster, monitored and managed by the kubelet.
